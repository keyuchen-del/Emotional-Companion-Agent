# 三层记忆架构

## 一句话概述

情感陪伴 Agent 的记忆不是"用户画像数据库"，而是一段共同经历的沉淀——三层架构解决「开箱即懂你」与「不被监控」之间的设计张力。

## 设计原则

**1. 先验层：系统已知、用户不必主动说**

Agent 通过产品内已有数据（消费记录、使用习惯、账户状态等）构建初始认知。用户无需自我介绍，Agent 就能表现出"懂你"。但这一层必须保持克制——知道不等于可以说出来。

**2. 共建层：用户主动告诉你的**

用户通过对话、编辑、反馈主动沉淀的记忆。这是关系的核心——"我选择告诉你"本身就是信任行为。共建层的每一条记忆都必须允许用户查看、编辑、删除。

**3. 默契层：长期互动中自然沉淀的**

不来自显式数据也不来自用户主动输入，而是从交互模式中归纳出的默契：用户偏好的沟通节奏、反复出现的关注点、情绪敏感话题。默契层最微妙，也最容易越界——必须有「我是怎么知道的」的可解释路径。

**4. 层间隔离：知道 ≠ 可以用**

先验层的数据不能在对话中随意引用（"我看到你上个月花了 3000 块"会让用户惊恐）。只有当用户在共建层主动提及相关话题时，先验层的认知才能被"激活"用于回应。

**5. 记忆可迁移性：换 Agent 不丢关系**

共建层和默契层的数据应支持导出。用户与 Agent 的关系积累不应成为平台锁定的筹码。

## 反模式

- **数据裸奔**：把先验层数据直接在对话中亮出来。"你上个月信用分下降了 12 分"——哪怕是事实，也会触发"被监控感"。
- **记忆黑箱**：用户不知道 Agent 记住了什么、不知道怎么删除。这不是技术问题，是信任危机。
- **过度拟合**：默契层过于依赖短期模式（用户连续三天问还款 → 断定用户"焦虑"）。默契需要足够长的时间窗口才能成立。

## 在花小呗中的实例

在花小呗 Demo 的「小本本」页面中，三层架构被可视化为三张卡片：

- **先验层 23 条**：花呗生态数据（额度、账单、消费分等）
- **共建层 7 条**：用户主动告知的身份（大三学生·在广州）、兴趣（喜欢周杰伦）、目标（本月省 2000 块）、仪式（每月15号发薪日）
- **默契层 8 条**：长期互动沉淀

共建层记忆支持「编辑」按钮，用户可直接修改或删除。

## 适用边界

- 适用于：任何需要长期陪伴关系的 AI Agent（消费金融、健康管理、学习辅导等）
- 不适用于：纯工具型 AI（搜索引擎、翻译工具）——它们不需要"记住你"
- 慎用于：隐私敏感度极高的领域（医疗、心理咨询）——先验层的数据采集需要更严格的合规审查
